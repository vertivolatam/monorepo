**Hallazgo central:** el mismo atributo lógico (un *bound*/setpoint de sensor) está realizado de forma
inconsistente en **5 superficies sin source-of-truth**: 3 tablas SQL de Serverpod (`crop_models`,
`greenhouses`, `growth_stage_definitions`), los JSON del Pi (`config/{current,defaults}/*.json`) y la
SQLite del crop_explorer (`setpoints`). Es el patrón `*_bio-in-5-tables` (1NF / DRY / multiple-writers-no-owner)
aplicado a los bounds agronómicos.

> **Nota metodológica (por qué el verificador determinístico subreporta):** `findDuplicatedColumns(sql, {minTables:2})`
> matchea por **nombre exacto** de columna. Acá cada superficie **renombra** el atributo
> (`idealPhMin` vs `phMin` vs `…Override` vs `nutrient_solution_ph_input_lower_bound` vs `setpoints.field='ph'`),
> así que el match léxico da casi vacío. El drift es **semántico**, confirmado por razonamiento LLM + anclas
> cross-store. Que los nombres estén deliberadamente diferenciados es, paradójicamente, evidencia de
> divergencia NO gobernada. Confidence = `confirmed`.

### SCH-004 — Temperature bounds duplicados + ambigüedad semántica · `high` · confirmed
- `crop_models.idealTemperatureMin/Max` — `definition.sql:167-168`
- `greenhouses.temperatureMin/Max` — `definition.sql:257-258` (`greenhouse.spy.yaml:14-15`)

**Agravante:** el JSON del Pi modela **temperatura de solución nutritiva**, mientras el SQL modela

físicas distintas**. El modelo canónico DEBE desambiguar (`air_temp` vs `solution_temp`). `cure_map: [ownership, ci_guard, hook]`.

### SCH-006 — EC/DO/ORP/TDS sin hogar canónico (drift inverso) · `medium` · confirmed
Estos bounds de solución nutritiva existen **solo en el JSON del Pi**, sin columna/tabla canónica en
Serverpod → no se pueden gobernar/editar desde el cloud:
- `nutrient_solution_{do,ec,orp,tds}_input_{lower,upper}_bound` — `apps/raspberry/config/current/indoor.json:6-11,14-15`

**Riesgo:** la app móvil/cloud no puede leer ni setear los parámetros más críticos de la nebuponía (EC/DO/ORP). El modelo canónico debe **incluirlos** (no solo unificar los que ya están en SQL). `cure_map: [ownership, ci_guard]`.

### SCH-007 — Múltiples escritores, sin owner canónico · `medium` · confirmed
Los bounds los escriben: el Pi (file edit / orquestador `src/config.py`), el crop_explorer (`crops.db` + `setpoint_audit`), y Serverpod (`greenhouse.update`, `cropCatalog.update`). **No hay CODEOWNERS ni dueño canónico** del concepto "setpoint/bound".
- Evidencia: ausencia de `CODEOWNERS` para el modelo; escritores en `apps/raspberry/src/config.py:45,73,114`, `crop_explorer/db.py` (origin/main), `apps/vertivo_server/lib/src/greenhouses/greenhouse_endpoint.dart` (update). `cure_map: [ownership, ci_guard, agent_rule, hook]`.

## Cure proposals (v1 — propuestas, NO auto-aplicadas)

### C1 · ownership — owner canónico del modelo de setpoints
Definir un único `setpoint`/`orchestrator_config` canónico (Postgres/Serverpod write model) y un CODEOWNERS:
```
# .github/CODEOWNERS (propuesta)
/apps/vertivo_server/lib/src/setpoints/        @lapc506
/apps/raspberry/src/config.py                  @lapc506
openspec/changes/**/setpoint*                  @lapc506
```

### C2 · ci_guard — falla si reaparece un bound fuera del canónico
```bash
# scripts/ci/guard-setpoint-drift.sh (propuesta)
# Falla si aparece un nuevo *Min/*Max/*_bound fuera de lib/src/setpoints o del JSON seed declarado.
rg -n '(Min|Max)\b|_(lower|upper)_bound' apps/vertivo_server/lib apps/raspberry/src \
  | rg -v 'lib/src/setpoints/|# canonical-setpoint-allowlist' \
  && { echo "❌ bound fuera del modelo canónico"; exit 1; } || exit 0
```

### C3 · agent_rule — regla atómica de ownership
Añadir a `.atomic-design-rules.json` (o `/make-no-mistakes:atomic-rules-init`) que todo bound/setpoint debe
pasar por el modelo canónico; los JSON del Pi pasan a ser **seed derivado**, no fuente.

### C4 · hook — PreToolUse guard
Hook que bloquee escrituras a `config/current/*.json` que cambien bounds sin metadata de provenance
(`mechanism`/`actor`/`reason`) — alinea con el device-shadow del change multinode (ADR-5).
